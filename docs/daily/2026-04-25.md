# 2026-04-25 Daily Change Log

## Goal

Investigate why 2026 transactions categorized while filtering to `Show Only Uncategorized` appeared uncategorized again the next day, then verify the app end to end and close out with local and remote `main` in sync.

## Root Cause

- The category saves themselves were working.
- The reset happened later when sync jobs re-upserted existing transaction rows.
- Plaid sync was sending Plaid's category into `public.transactions.category`.
- Amazon Business sync was sending blank `category`, blank `splits`, and blank review fields into `public.transactions`.
- Because those sync paths use upsert conflict keys, rerunning sync could overwrite user-reviewed accounting state after the user had already cleared the uncategorized queue.

## Sync Preservation Fix

- Updated Plaid sync row construction so Plaid category data is no longer written into the app's reviewed accounting category field.
- Updated Amazon Business transaction row construction so sync reruns no longer send:
  - `category: ""`
  - `splits: []`
  - blank `review_status`
  - blank `deduction_status`
  - blank `review_note`
- Kept sync metadata updates intact, including date, amount, merchant/name, source payload, Amazon order IDs, and line-item keys.
- Added comments at both sync row builders to make clear that category, split, and review fields are user-reviewed state and should not be clobbered by future sync changes.

## Frontend Save Hardening

- Strengthened `Api.getCurrentUserId()` to validate/refresh the Supabase session before saves.
- Kept update/delete writes scoped to the signed-in user's row through `id` plus `user_id`.
- Added row-count verification after transaction updates/deletes so silent RLS misses surface as save failures.
- Added a fallback retry path for optional review fields if the deployed DB schema/cache is stale.
- Improved split save handling:
  - prevents concurrent split saves,
  - rolls back UI state if persistence fails,
  - rolls back remove/edit split operations if persistence fails,
  - keeps the transaction category aligned to `SPLIT` whenever splits exist.

## Audit Fixes From Deep Dive

- Search result selection now actually applies a scoped search filter instead of only jumping to the clicked row.
- Clearing search now updates totals/progress before re-rendering.
- Transaction type filter now appears in the date/filter summary, for example `Year: 2026 • Uncategorized`.
- Reloads with zero visible rows now render the correct `No Transactions` / `No Transactions Found` states.
- Receipt removal now clears the DB receipt pointer before storage cleanup, avoiding broken DB references if storage deletion fails.
- Deleting a transaction now also attempts to clean up the attached receipt storage object.
